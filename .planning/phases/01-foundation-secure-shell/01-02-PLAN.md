## Objective

Replace the Walking Skeleton's raw `CREATE TABLE IF NOT EXISTS` with the production-grade SQLite stack: versioned Drizzle migrations that auto-run at startup, WAL pragmas with foreign keys, `integrity_check` on launch, rotating database backups (last 10), and `app.requestSingleInstanceLock()` BEFORE any DB code opens (prevents the WAL-corruption race in Pitfall #8). Migration files are placed in `resources/migrations/` and wired into `electron-builder.yml` as `extraResources` so they're accessible from packaged builds (the ASAR migration footgun from Pitfall #3).

After this plan, FOUND-02 is fully satisfied: campaign data persists in SQLite with versioned schema migrations.

## Context

Locked decisions implemented here:
- **D-16:** Full SQLite safety stack — WAL mode (`journal_mode=WAL; synchronous=NORMAL`), `integrity_check` on launch, backup rotation (copy to `solocampaign-backup-{timestamp}.db` before each session open, keep last 10)
- **D-17:** Migrations auto-run at startup via Drizzle `migrate()`; SQL files in `extraResources` so they're accessible in packaged builds
- **D-18:** `app.requestSingleInstanceLock()` runs BEFORE `app.whenReady()`; on second-instance attempt, focus and restore the existing window (no secondary notification)

Research pins (per RESEARCH.md):
- §4 "Drizzle ORM + auto-run migrate()" — the ASAR footgun + `app.isPackaged ? process.resourcesPath : __dirname` pattern (Pitfall #3)
- §10 "SQLite Safety Stack" — pragma order, integrity_check result handling, backup rotation logic
- §9 "Single-Instance Lock" — must run BEFORE app.whenReady, second-instance event handler restores + focuses
- Pitfall #8 — open DB AFTER lock confirmed
- Pitfall #9 — integrity_check returning non-'ok' is logged via electron-log (no UI in Phase 1)
- "Backup rotation trigger" Open Question #4 — Phase 1 backs up on app launch (only meaningful trigger that exists yet)

<threat_model>

| Threat | STRIDE | Mitigation in this plan |
|--------|--------|------------------------|
| T-DB-01: WAL corruption from second-instance race | T (Tampering) | `app.requestSingleInstanceLock()` runs BEFORE `app.whenReady()`; if lock not acquired, `app.quit()` immediately — DB code never runs in the second process. Second-instance event focuses + restores the existing window. |
| T-DB-02: Drizzle migration files not found in packaged build → app crashes at startup | T/D | Migration files in `resources/migrations/`, wired into `electron-builder.yml` `extraResources`; path resolution uses `app.isPackaged ? process.resourcesPath : __dirname` per Pitfall #3 |
| T-DB-03: Silent DB corruption (page-level damage from crash/disk error) | I (Information Disclosure — data loss) | `PRAGMA integrity_check` runs after every DB open; non-'ok' result logged at ERROR level via electron-log; backup rotation preserves last 10 snapshots so a future Phase 2 task can restore |
| T-DB-04: Schema drift across versions (user upgrades app, schema is now newer than DB) | T | Drizzle `migrate()` runs all pending migrations idempotently at every startup |
| T-DB-05: Backup directory grows unbounded → disk exhaustion | D (Denial of Service) | Backup rotation deletes oldest beyond 10 in `backupRotation.ts` |

</threat_model>
