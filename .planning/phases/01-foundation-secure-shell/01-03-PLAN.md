## Objective

Implement `SecretStorageService` — a thin, well-tested wrapper around Electron's `safeStorage` API — and expose the `secrets` tRPC verbs (`exists`, `set`, `delete`) for Phase 3 to consume. **No `get` procedure is exposed over IPC**: returning plaintext secrets to the renderer over an IPC channel would defeat the purpose of safeStorage. Phase 3 will design a per-request scoped use pattern (e.g., `chat.send` accepts an opaque secret key, the main process decrypts in-place, makes the LLM call, and never returns the plaintext to the renderer).

The service handles the headless-Linux fallback per Pitfall #10: when `safeStorage.getSelectedStorageBackend()` returns `basic_text` (which is just a hard-coded plaintext password masquerading as encryption), the service base64-encodes the value with a documented `B64:` prefix and logs a warning. The UI warning surface is Phase 3's job.

After this plan, FOUND-04 is fully satisfied at the architecture level. The UI surface that uses it lands in Phase 3.

## Context

Locked decisions implemented here:
- **D-15:** Phase 1 ships the full `SecretStorageService` (`encrypt`, `decrypt`, `exists`, plus `remove` — added because tRPC needs a delete verb). Phase 3 imports it and adds the provider config UI. Headless Linux fallback: when `safeStorage.isEncryptionAvailable()` is false OR backend is `basic_text`, log warning and fall back to base64.

Research pins:
- §6 "SecretStorageService (safeStorage wrapper)" — full class body including `isSecure()` check that combines `isEncryptionAvailable() &amp;&amp; getSelectedStorageBackend() !== 'basic_text'`
- §6 — filesystem key normalization (`key.replace(/[^a-zA-Z0-9_.-]/g, '_')`) to prevent path traversal
- §6 — `B64:` magic prefix discriminates fallback ciphertexts from real safeStorage payloads
- Pitfall #10 — `isEncryptionAvailable()` returns `true` on Linux even when the backend is the trivial `basic_text` fallback; the service MUST check both
- Security Domain V6 Cryptography — never hand-roll a cipher; safeStorage is the only crypto primitive
- "Note: NO `get` procedure that returns plaintext over IPC" (RESEARCH §2 tRPC router)

This plan does NOT modify the global tRPC router file (`src/main/trpc/router.ts` — owned by 01-01). It only edits `src/main/trpc/routers/secrets.ts`, which 01-01 left as an empty `t.router({})` placeholder. This is what makes 01-03/04/05 safe to run in parallel within Wave 2.

<threat_model>

| Threat | STRIDE | Mitigation in this plan |
|--------|--------|------------------------|
| T-KEY-01: Plaintext API key leaks to renderer process via IPC | I (Information Disclosure) | NO `get` procedure exposed on `secrets` router. Only `exists`, `set`, `delete`. Phase 3 will design a scoped-use pattern that decrypts in main, uses, and never returns plaintext. |
| T-KEY-02: Plaintext key written to disk via headless-Linux `basic_text` "encryption" | I | `isSecure()` checks BOTH `isEncryptionAvailable()` AND `getSelectedStorageBackend() !== 'basic_text'`. When false, value is base64-encoded with a `B64:` prefix and `electron-log` warns at WARN level so the operator knows. UI warning surface deferred to Phase 3 per D-15. |
| T-KEY-03: Path traversal via malicious key name → arbitrary file write | T (Tampering) | Key string is regex-normalized (`/[^a-zA-Z0-9_.-]/g → '_'`) BEFORE being concatenated into the filepath. The normalized name cannot contain `/`, `\`, `..`, or null bytes. |
| T-KEY-04: Renderer fakes a `secrets.set` with absurd key/value sizes → disk exhaustion | D (DoS) | Zod schema enforces `key: z.string().min(1).max(64).regex(/^[a-zA-Z0-9_.-]+$/)` and `value: z.string().min(1).max(2048)` (API keys/tokens fit easily under 2KB) |
| T-KEY-05: Secret values logged accidentally by electron-log | I | `secretStorageService.ts` and the tRPC router file MUST NOT log the `value` parameter. Pre-commit grep gate: no `console.log` or `log.info` calls in either file that include the `value` argument |

</threat_model>

    - The service's filesystem operations are async (`fs/promises`); the API surface is `Promise`-based
    - The service does NOT throw when the secrets directory doesn't exist on first call to `init()` — it creates it via `mkdir({recursive: true})`

  <action>
    Create `src/main/secrets/secretStorageService.ts` implementing the `SecretStorageService` class per RESEARCH §6 (use that structure as the reference — fields `dir`, methods `init`, `isSecure`, `encrypt`, `decrypt`, `exists`, `remove`, private `filepath`). Use `import { safeStorage, app } from 'electron'` and `import { readFile, writeFile, unlink, mkdir } from 'fs/promises'` and `import { existsSync } from 'fs'`.

    `isSecure()` MUST check BOTH `safeStorage.isEncryptionAvailable()` AND `safeStorage.getSelectedStorageBackend() !== 'basic_text'` (Pitfall #10 — `isEncryptionAvailable` returning true on Linux is not enough). When false, log warning via `electron-log` ONCE per process (cache the warning state in a module-scope boolean so it doesn't spam).

    `encrypt(key, value)` path: if `isSecure()` → write `safeStorage.encryptString(value)` buffer to `filepath(key)`. Else → write `Buffer.concat([Buffer.from('B64:'), Buffer.from(Buffer.from(value).toString('base64'))])` to `filepath(key)`. Use `await writeFile(filepath, buf)` — do NOT use `writeFileSync`.

    `decrypt(key)`: if file doesn't exist, return `null`. Read file. If first 4 bytes are the ASCII `B64:`, slice and base64-decode. Else, `safeStorage.decryptString(buf)`.

    `filepath(key)`: normalize via `key.replace(/[^a-zA-Z0-9_.-]/g, '_')` BEFORE concatenating with `this.dir`. Use `path.join(this.dir, normalized + '.enc')`.

    Export a module-level singleton via `src/main/secrets/index.ts`: `export const secretStorage = new SecretStorageService()`. The main process awaits `secretStorage.init()` once at startup (do NOT call init from inside `encrypt`/`decrypt` — that adds a `existsSync` check per call. One init at startup is enough).

    The service does NOT log secret values, ever. The only allowed log statement in this file involves the fallback warning, which says "safeStorage backend is basic_text — values are not securely encrypted." It does NOT include the key or value.
