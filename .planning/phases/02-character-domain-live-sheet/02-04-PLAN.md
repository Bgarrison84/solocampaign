<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| renderer → main (importImage trigger) | Renderer triggers via tRPC mutation but supplies NO file path — main process uses OS dialog (per Pitfall 4 / T-PATH) |
| OS filesystem ↔ main (jimp read + fs write) | Source path comes from OS dialog (user-confirmed); destination path is derived inside main from app.getPath('userData') — never from renderer |
| main → renderer (base64 data URL) | Image bytes cross IPC as base64 — payload bounded by 1024px resize (~50-200KB JPEG); CSP allows data: per Phase 1 |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-09 | Tampering | Path traversal via importImage | mitigate | Source path comes from dialog.showOpenDialog (OS-validated); destination derived from app.getPath('userData') + campaignId (uuid) + sanitized basename; never accepts renderer-supplied path per Pitfall 4 |
| T-02-10 | Information disclosure | getPortraitDataUrl / getCoverDataUrl arbitrary file read | mitigate | Renderer passes characterId/campaignId, not path; main reads stored relativePath from DB (controlled at write time by importImage); path is always under app.getPath('userData')/images/ |
| T-02-11 | Denial of service | Huge source image OOM | mitigate | jimp resize to MAX_DIMENSION=1024 enforced before write; jimp streams images, doesn't load full pixel buffer for resize; further mitigation: filter extensions in dialog |
| T-02-12 | Tampering | CSP bypass via file:// | mitigate | Image display uses base64 data URL only; CSP unchanged; per Pitfall 4 file:// is never used in img src |
</threat_model>
