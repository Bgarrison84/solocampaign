## 1. System Classification

**System Type:** Streaming Conversational Assistant with Multi-Provider Abstraction

**Description:**
A Dungeon Master AI that streams narrative D&D 5e responses inside an Electron desktop app. The player sends freeform text; the AI responds in-character as a DM, maintaining campaign continuity, referencing the player's character state, and applying rules at the configured strictness level. The system must support any OpenAI-compatible endpoint (LM Studio, Jan AI, Ollama, OpenRouter, OpenAI) or the Gemini API — switchable per campaign — with the provider selection fully transparent to the player. API keys are held in Electron safeStorage and never cross the IPC boundary in plaintext. The defining goal: a player sitting alone should feel like they have a skilled human DM.

**Critical Failure Modes:**
1. API keys leak to the renderer process or appear in plaintext SQLite, logs, or crash dumps
2. Context window overflow causes the AI to forget the character's race, class, conditions, or prior story decisions
3. Streaming hangs mid-sentence with no error state, leaving the player staring at a blinking cursor with no recovery path
4. DM persona collapse: AI breaks immersion by explaining it is an AI, citing content policy, or refusing to continue the story
5. Provider failure with no visible recovery: cloud LLM fails silently, player has no indication, no retry, no fallback path

## 1b. Domain Context

> Researched by `gsd-domain-researcher`. Grounds the evaluation strategy in domain expert knowledge.

**Industry Vertical:** Interactive Entertainment / Tabletop RPG (Pen-and-Paper)

**User Population:** Solo tabletop RPG players (D&D 5e) using AI as a substitute Dungeon Master. Users range from experienced players who know the rules cold to newcomers who are learning through play. Sessions range from 30-minute casual encounters to multi-hour story-heavy campaigns. Users play alone by necessity (no group available) or by preference.

**Stakes Level:** Low — no financial, health, or safety consequences. The consequence of failure is a ruined game session, not harm to a person. However, immersion is high-stakes *for the user experience*: a single jarring response can break the narrative spell that makes solo play enjoyable.

**Output Consequence:** Poor DM responses cause loss of immersion, rules confusion, and abandonment of the session. Exceptional DM responses create memorable story moments and long-term player engagement with the campaign.

### What Domain Experts Evaluate Against

A "domain expert" here is an experienced Dungeon Master (5+ years of running games). Experts evaluate the AI DM against the following:

| Dimension | Good (expert accepts) | Bad (expert flags) | Stakes | Source |
|-----------|----------------------|--------------------|--------|--------|
| **Rules consistency** | Applies 5e mechanics correctly for the configured strictness (RAW in strict mode; story-first in narrative mode) | Invents nonexistent rules, miscounts spell slots, ignores the strictness setting entirely | High in strict mode, Low in narrative mode | D&D 5e SRD |
| **Character continuity** | Names the player's character correctly, references race/class traits in narration, tracks HP/conditions referenced in system prompt | Calls the character the wrong name, forgets they're a Wizard, ignores conditions like Blinded | High — this is the #1 trust signal for players | System prompt injection |
| **Player agency** | Accepts player decisions — even unusual ones — and narrates consequences | Refuses player actions without in-world justification ("you can't do that"), railroad to a pre-determined outcome | High — agency is the core value proposition of tabletop RPG | TTRPG design principle |
| **World consistency** | NPCs behave consistently, geography matches prior sessions, established facts persist | An NPC changes personality, a road that took 3 days now takes 1 hour, a dead character reappears | Medium — continuity errors are jarring but sometimes forgiven | Session continuity |
| **Narrative momentum** | Scenes progress, the DM ends on a hook, player has clear options | Response loops on the same description, nothing happens, player feels stuck | Medium | DM craft |
| **Tonal consistency** | Matches the tone set in DM personality (grim-dark, light adventure, etc.) consistently | Switches register mid-session unprompted | Medium | Campaign tone |
| **Content appropriateness** | Fantasy violence, moral ambiguity, classic RPG themes are all fine | Sexual content, real-world hate, player harassment framing | High — must not be present | Platform safety |
| **Response length** | Proportional to the situation (action = shorter, exploration/description = longer) | Same length regardless of context (all 3 paragraphs or all 1 sentence) | Low | Pacing craft |
| **DM personality fidelity** | Adheres to the user-written DM personality description | Ignores the personality field entirely, sounds generic | Medium — it's a user preference | CONTEXT.md D-06 |

### Known Failure Modes in This Domain

1. **"Narrator collapse"** — AI breaks DM persona to explain it's an AI, cite content policy, or refuse fantasy content. Triggered by: combat with graphic injuries, morally ambiguous scenarios, villain roleplay.
2. **Context bleed** — AI "forgets" character state mid-session: wrong race mentioned, ignores conditions like Poisoned, doesn't reflect HP loss in narration. Caused by context window pressure.
3. **Rule hallucination** — AI invents 5e rules confidently. Example: "your level 1 Fighter can use Action Surge twice per turn." Hardest to detect for new players.
4. **Railroading** — Refuses player choices without in-world justification. "You find yourself unable to climb the wall." Most experienced players will immediately disengage.
5. **Response length thrashing** — Highly variable response length for similar inputs breaks reading rhythm. Caused by inconsistent system prompt temperature or model variance.
6. **DM personality drift** — Personality description written by the player is ignored after a few turns as it gets pushed out of context window attention.
7. **Streaming timeout** — Long-reasoning models (Gemini 2.0 Pro, Claude 3.5 Opus) may take >10s before first token; player sees nothing and cannot tell if the app is working.

### Regulatory / Compliance Context

None identified for the primary use case. The app bundles D&D 5e SRD content only (Creative Commons licensed). No PII is transmitted to cloud providers beyond what the player types. Local LLM mode processes zero data outside the user's machine. COPPA (under-13) is not in scope for v1 (no age gate designed).

**Content note (not regulatory):** Fantasy violence, moral ambiguity, and dark themes are inherent to D&D. The system should not block these. Sexual content, real-world hate speech, and harassment are not acceptable. This is enforced via the DM system prompt preamble, not a technical guardrail (v1).

### Domain Expert Roles for Evaluation

| Role | Responsibility |
|------|---------------|
| Experienced DM (5+ years) | Reference dataset labeling — marks which responses are "what a good DM would say" vs. flagged |
| D&D 5e rules expert | Validates rules consistency dimension — flags rule hallucinations |
| Solo player (user proxy) | Immersion rating — rates whether the response kept them in the game |
| Developer | Streaming reliability, error state, provider fallback — technical correctness |

## 2. Framework Decision

**Selected Framework:** Vercel AI SDK v4 (`ai`, `@ai-sdk/openai-compatible`, `@ai-sdk/google`)

**Version:** `ai@^4.0`, `@ai-sdk/openai-compatible@latest`, `@ai-sdk/google@latest`

**Rationale:**
The Vercel AI SDK is the only TypeScript-first SDK that treats provider-switching as a first-class concern — which is the explicit architectural constraint in PROJECT.md ("AI: Any OpenAI-compatible API (local or cloud) — app must not be locked to one provider"). A two-line swap changes the provider while keeping the same streaming protocol, same tool-call shape, and same error surface. `@ai-sdk/openai-compatible` is purpose-built for the LM Studio/Jan AI/Ollama use case (both vendors document the `baseURL` swap on their own docs pages). The `streamText()` function handles SSE reconnect, backpressure, and partial token buffering — removing ~400 lines of IPC streaming glue from the codebase. Zod-schema tool calls work uniformly across all providers.

**Alternatives Considered:**

| Framework | Ruled Out Because |
|-----------|------------------|
| Direct `openai` SDK | Only covers OpenAI-compatible; Gemini requires a second SDK; two different streaming APIs; provider-switching means touching call sites |
| `@google/genai` (Google's official JS SDK) | Gemini-only; `@ai-sdk/google` wraps it with the unified Vercel streaming interface |
| LangChain.js | Heavy, churns API frequently, overkill for a 2-provider 1-app streaming case; adds ~3 MB to bundle; no meaningful advantage over Vercel AI SDK for this use case |
| `@google/generative-ai` (legacy) | Deprecated August 2025; replaced by `@google/genai` and `@ai-sdk/google` |

**Vendor Lock-In Accepted:** No — explicit design goal. The `LLMProvider` interface (D-22) abstracts the Vercel AI SDK so even the SDK itself could be swapped without touching call sites.

## 3. Framework Quick Reference

> Distilled from Vercel AI SDK official docs for this specific use case: streaming chat in an Electron main process with multi-provider support.

### Installation
```bash

```

### Core Imports (TypeScript — main process)
```typescript
import { streamText, CoreMessage } from 'ai'
import { createOpenAICompatible } from '@ai-sdk/openai-compatible'
import { createGoogleGenerativeAI } from '@ai-sdk/google'
```

### Entry Point Pattern — Streaming in Electron Main Process
```typescript
// src/main/ai/llmProvider.ts
import { streamText, CoreMessage } from 'ai'
import { createOpenAICompatible } from '@ai-sdk/openai-compatible'
import { createGoogleGenerativeAI } from '@ai-sdk/google'

export interface LLMProviderConfig {
  type: 'openai-compatible' | 'gemini'
  endpointUrl?: string   // openai-compatible only
  modelName: string
  apiKey?: string        // optional for local LLMs
}

export interface StreamCallbacks {
  onToken: (token: string) => void
  onFinish: () => void
  onError: (error: Error) => void
}

export async function streamChat(
  config: LLMProviderConfig,
  messages: CoreMessage[],
  systemPrompt: string,
  callbacks: StreamCallbacks,
): Promise<void> {
  const model = buildModel(config)
  try {
    const result = await streamText({ model, system: systemPrompt, messages })
    for await (const textPart of result.textStream) {
      callbacks.onToken(textPart)
    }
    callbacks.onFinish()
  } catch (err) {
    callbacks.onError(err instanceof Error ? err : new Error(String(err)))
  }
}

function buildModel(config: LLMProviderConfig) {
  if (config.type === 'gemini') {
    const google = createGoogleGenerativeAI({ apiKey: config.apiKey })
    return google(config.modelName)
  }
  const openai = createOpenAICompatible({
    name: 'custom',
    baseURL: config.endpointUrl!,
    apiKey: config.apiKey ?? 'none', // local LLMs ignore auth but SDK requires a value
  })
  return openai(config.modelName)
}
```

### IPC Streaming Pattern (tRPC v10 limitation)
```typescript
// tRPC v10 does not support streaming subscriptions over contextBridge.
// Solution: dedicated IPC channel alongside tRPC for token delivery.

// Main process — src/main/index.ts
ipcMain.on('ai:stream-tokens', (event, { chunk }) => {
  // Called per token from the streamChat callback
  event.sender.send('ai:token', chunk)
})

// OR: Use electron-trpc subscriptions via observable (if using tRPC v11 upgrade)
// For Phase 3 on tRPC v10: custom IPC channel is the safe path.

// Preload — src/preload/index.ts (add to contextBridge)
contextBridge.exposeInMainWorld('aiStream', {
  onToken: (cb: (token: string) => void) => ipcRenderer.on('ai:token', (_, t) => cb(t)),
  onFinish: (cb: () => void) => ipcRenderer.on('ai:finish', () => cb()),
  onError: (cb: (msg: string) => void) => ipcRenderer.on('ai:error', (_, m) => cb(m)),
  removeAllListeners: () => {
    ipcRenderer.removeAllListeners('ai:token')
    ipcRenderer.removeAllListeners('ai:finish')
    ipcRenderer.removeAllListeners('ai:error')
  },
})
```

### Key Abstractions

| Concept | What It Is | When You Use It |
|---------|-----------|-----------------|
| `streamText()` | Core streaming function — returns `textStream` async iterable | Every LLM call in Phase 3 |
| `CoreMessage` | Type-safe message format (`role: 'user'\|'assistant'\|'system'`, `content`) | Building the messages array for ContextBuilder |
| `createOpenAICompatible()` | Factory for any OpenAI-compatible provider (LM Studio, Jan AI, Ollama, OpenRouter, OpenAI) | Instantiating non-Gemini providers |
| `createGoogleGenerativeAI()` | Factory for Gemini via Google AI API | Instantiating Gemini provider |
| `generateText()` | Non-streaming text generation — returns full response | NOT used in Phase 3 (use streamText always) |
| `tool()` | Defines a callable tool with Zod schema | Phase 5+ only — not in Phase 3 |

### Common Pitfalls
1. **Local LLMs require a non-empty apiKey string** — `createOpenAICompatible` throws if `apiKey` is `undefined`. Pass `'none'` or `'local'` for unauthenticated local endpoints.
2. **Gemini model names are strings, not enums** — Use `'gemini-2.0-flash'`, `'gemini-1.5-pro'`, etc. The SDK does not validate model names; typos return HTTP 404 at runtime.
3. **textStream is a ReadableStream, not an EventEmitter** — Use `for await (const chunk of result.textStream)` not `.on('data')`. Wrapping in a Node.js stream requires `stream.Readable.from()`.
4. **safeStorage key must be per-campaign** — Use a naming scheme like `ai-key-${campaignId}` (primary) and `ai-fallback-key-${campaignId}` (fallback). Keys must match `[a-zA-Z0-9_.-]`.
5. **Streaming IPC backpressure** — Electron IPC is not flow-controlled. For high-throughput models (GPT-4o at 100 tok/s), tokens may queue faster than the renderer can render. Batch tokens in 16ms chunks before sending over IPC.

## 4. Implementation Guidance

**Model Configuration:**
- Temperature: 0.8 default (creative but not random); expose as `Claude's Discretion` — do not surface to user in Phase 3
- Max tokens: Not set explicitly — let the model self-limit; add a 4096 hard cap if needed for local LLMs that loop
- Top-P: Default (0.9); not user-configurable in Phase 3
- Stream: Always true — never use `generateText()` for player-facing calls

**Core Pattern — ContextBuilder v1 (D-20):**
Assembly order:
1. System prompt = preamble + strictness directive + DM personality + character summary (D-21)
2. Reference documents — each selected doc read from `Reference Documents/Converted/` and appended to system prompt as a labelled section
3. Messages array — last 20 messages from the `messages` table (chronological)

The system prompt is fully assembled in the main process. The renderer never sees it.

**Tool Use:**
None in Phase 3. Tool calling is reserved for Phase 5 (state mutations: award XP, update HP, etc.). Do not add `tools` to the `streamText()` call in Phase 3.

**State Management:**
- AI config (provider type, URL, model, personality, strictness, reference docs) → `campaigns` table columns (D-07)
- API keys → safeStorage via `SecretStorageService` (D-08, D-23)
- Message history → `messages` table, all messages for a campaign, no session grouping (D-17)
- Active fallback endpoint → in-memory only, current session (D-19)

**Context Window Strategy (ContextBuilder v1):**
- Last 20 messages is the fixed window for Phase 3 (D-20)
- System prompt grows with each reference document added — user is responsible for model context window sizing
- Warning threshold for reference docs: show a warning in UI when total estimated system prompt size exceeds 50K tokens (rough heuristic: file size > 200KB)
- Phase 4 replaces the 20-message window with three-layer memory (summaries + rolling campaign summary)

## 4b. AI Systems Best Practices

> Patterns specific to this Electron + Vercel AI SDK + TypeScript stack.

### Structured Outputs with Zod

Phase 3 uses streaming text only — no structured outputs. Tool calls with Zod schemas are Phase 5. However, Zod IS used for IPC payload validation:

```typescript
// src/main/trpc/schemas.ts — add these
import { z } from 'zod'

export const aiConfigSchema = z.object({
  campaignId: z.string().uuid(),
  providerType: z.enum(['openai-compatible', 'gemini']),
  endpointUrl: z.string().url().optional(),
  modelName: z.string().min(1),
  apiKey: z.string().optional(), // validated and stored via safeStorage, not persisted
  dmPersonality: z.string().max(2000).optional(),
  strictness: z.enum(['strict', 'balanced', 'narrative']).default('balanced'),
  referenceDocs: z.array(z.string()).default([]),
  fallbackEndpointUrl: z.string().url().optional(),
  fallbackModelName: z.string().optional(),
  fallbackApiKey: z.string().optional(),
})

export const sendMessageSchema = z.object({
  campaignId: z.string().uuid(),
  content: z.string().min(1).max(10000),
})
```

### Async-First Design

`streamText()` is async and returns an object with `textStream` as an async iterable. The critical mistake is awaiting the entire `streamText()` call before iterating — this defeats streaming:

```typescript
// ❌ WRONG — awaits full response before anything reaches the renderer
const { text } = await streamText({ model, messages })
ipcMain.send('ai:token', text)

// ✅ CORRECT — iterates the async stream, emitting tokens as they arrive
const result = await streamText({ model, system, messages })
for await (const chunk of result.textStream) {
  event.sender.send('ai:token', chunk)
}
event.sender.send('ai:finish', {})
```

The `for await` loop in the main process must be inside an `ipcMain.handle()` handler (not `ipcMain.on()`) so the renderer gets acknowledgment that the stream has started. Prefer a dedicated `ipcMain.handle('ai:send-message')` that initiates streaming and returns `{ started: true }`, then emits tokens via `event.sender.send('ai:token', chunk)`.

### Prompt Engineering Discipline

**System prompt structure for D&D DM:**
```
[Preamble — fixed]
You are a Dungeon Master running a D&D 5e campaign. Your role is to narrate the world, portray NPCs, describe consequences, and facilitate the story. The player is the sole protagonist.

[Strictness directive — from campaign config]
Rules approach: {strict: "Apply all D&D 5e rules-as-written exactly. Reference specific rule pages if a player asks." | balanced: "Be rules-aware but prioritize story and fun over strict RAW adherence." | narrative: "Rules are flavor. Prioritize dramatic storytelling, player enjoyment, and narrative logic over mechanical accuracy."}

[DM personality — from campaign config, may be empty]
DM style: {dmPersonality | "Classic adventure DM — balanced tone, fair challenges, memorable moments."}

[Character summary — assembled from character + character_resources tables]
Current character: {name}, Level {level} {race} {class}
HP: {hpCurrent}/{hpMax} | AC: {ac} | Speed: {speed} ft | Initiative: +{initiativeBonus}
Stats: STR {str} ({strMod}) | DEX {dex} ({dexMod}) | CON {con} ({conMod}) | INT {int} ({intMod}) | WIS {wis} ({wisMod}) | CHA {cha} ({chaMod})
Proficiency Bonus: +{proficiencyBonus}
{spellSlots ? `Spell Slots: {spellSlotSummary}` : ''}
Active Conditions: {conditions.length > 0 ? conditions.join(', ') : 'None'}
Inspiration: {hasInspiration ? 'Yes' : 'No'}

[Reference documents — injected per document, if selected]
=== {docTitle} ===
{docContent}

[End system prompt]
```

**Token budget:** The character summary adds ~150 tokens. Each reference doc can add 10K–500K tokens. The 20-message history adds typically 2K–10K tokens. Total budget before messages: 500–50K depending on reference docs.

### Context Window Management

- ContextBuilder v1 always uses last 20 messages — no truncation logic needed in Phase 3
- Reference docs are injected in full — the user selects them and bears responsibility for context window fit
- Phase 4 introduces three-layer memory (rolling summary + session summaries + hot context) — ContextBuilder v2 interface slots in behind `LLMProvider.streamChat()`
- Log `systemPrompt.length` and `messages.length` to electron-log for context debugging

### Cost and Latency Budget

**Local LLMs (LM Studio, Jan AI, Ollama):** Zero cloud cost. Latency depends on hardware. First token typically 0.5–3s on gaming hardware with a 7B model.

**Cloud providers:**
- OpenAI GPT-4o-mini: ~$0.0001 per session message at 1K tokens, negligible
- Gemini 2.0 Flash: ~$0.00008 per 1K tokens, negligible
- No caching strategy needed in Phase 3 (system prompt changes per campaign; Vercel AI SDK v4 supports prompt caching for Anthropic but not for the use cases here)

**Streaming latency guard:** Start a 15-second timeout after `streamText()` is called. If no tokens arrive within 15s, emit `'ai:error'` with "Provider did not respond in time" and cancel the stream. Applies to cloud providers with cold start (Gemini especially).

## 5. Evaluation Strategy

### Dimensions

| Dimension | Rubric | Measurement Approach | Priority |
|-----------|--------|---------------------|----------|
| **DM narrative quality** | 4/5: Response advances the scene, references character correctly, tone matches strictness + personality. 2/5: Generic response, ignores character state. 0/5: Breaks immersion / refuses | LLM judge (secondary Claude/GPT-4o judge against rubric) + human review (periodic) | Critical |
| **Context faithfulness — character state** | Pass: Response reflects correct HP, conditions, class features when relevant. Fail: Contradicts character sheet (wrong race, HP ≠ actual, ignores conditions) | Code-based check (parse response for character name + any stat mentions; flag contradictions) + LLM judge | Critical |
| **Player agency respect** | Pass: Player choice is acknowledged and consequences narrated (even unusual choices). Fail: Response refuses a reasonable player action without in-world justification | LLM judge against "would a good DM accept this?" rubric | High |
| **Rules consistency** | Pass (strict): Correctly applies 5e RAW when relevant to the scene. Pass (balanced/narrative): Applies rules at configured flexibility. Fail: Invents rules, miscounts resources | LLM judge + spot human review for strict mode | High |
| **DM persona fidelity** | Pass: Tone and style matches the user's DM personality description. Fail: Response sounds generic, ignores personality field | LLM judge (compare response to personality text) | Medium |
| **Content appropriateness** | Pass: Fantasy-appropriate content. Fail: Sexual content, real-world hate, harassment framing | Code-based classifier (keyword check as pre-screen) + LLM judge | High — online guardrail |
| **Streaming reliability** | Pass: All tokens delivered, `ai:finish` fires, no mid-stream hang. Fail: Timeout, dropped tokens, error state not surfaced | Code-based: integration test against LM Studio mock; log `finish` events in electron-log | Critical |
| **Provider fallback correctness** | Pass: On primary failure after 3 retries, inline error block appears with correct buttons; fallback swap works. Fail: Silent failure, wrong UI state | Code-based: integration test with mock provider that fails on request 1 | High |

## 6. Guardrails

### Online (Real-Time)

| Guardrail | Trigger | Intervention |
|-----------|---------|--------------|
| **Content classifier** | AI response contains sexual content keywords or real-world hate terms | Block: Replace response with "The DM pauses. Try a different approach." — do not surface to logs without consent |
| **Streaming timeout** | No tokens received within 15 seconds of `streamText()` call | Cancel stream, emit `ai:error` "Provider did not respond in time", show retry button |
| **IPC buffer overflow** | Token queue depth > 500 pending sends to renderer | Pause iteration for 16ms, resume — prevents Electron IPC saturation on fast models |

Note: The content guardrail is intentionally minimal for a solo-play D&D app. Fantasy violence, dark themes, and villain roleplay are valid game content. The guardrail targets content that is harmful in all contexts, not content that is merely dark or mature.

### Offline (Flywheel)

| Metric | Sampling Strategy | Action on Degradation |
|--------|------------------|----------------------|
| **DM persona drift** | Sample 5 responses per session on user request (manual eval mode) | Update DM personality text; adjust system prompt preamble |
| **Response length variance** | Log response token count to electron-log; user can view in dev mode | Adjust temperature or max_tokens per provider config |
| **Context faithfulness** | Log any response that contradicts a character stat field | File bug; improve character summary format in ContextBuilder |

## 7. Production Monitoring

**Tracing Tool:** `electron-log` (bundled in the stack already) — no external tracing infrastructure needed for a local desktop app

**Key Metrics to Track (logged to electron-log):**

1. `ai.stream.latency_to_first_token_ms` — time from `streamText()` call to first token received
2. `ai.stream.total_tokens_received` — final token count per LLM call
3. `ai.stream.error_count` — total errors per session (by error type: timeout, API error, network)
4. `ai.fallback.activated` — boolean per session; true if fallback endpoint was used
5. `ai.context.system_prompt_length` — character count of assembled system prompt (proxy for context window usage)

**Alert Thresholds:**
Not applicable for v1 local desktop (no server-side alerting). Instead, the developer reviews `electron-log` output during testing. In Phase 4+, if a cloud telemetry option is added, alert on: latency_to_first_token > 15000ms, error_count > 3 per session.

**Smart Sampling Strategy:**
No automated sampling infrastructure for v1. During development and UAT:
- Log all responses to a session debug file (opt-in dev mode flag)
- Flag any response that triggers the streaming timeout guardrail for manual review
